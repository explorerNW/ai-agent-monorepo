# Service Worker Implementation Guide

## Overview

This project implements a comprehensive Service Worker solution using a custom Vite plugin for build-time optimization. The Service Worker provides offline support, intelligent caching strategies, and automatic updates.

## Architecture

### Components

1. **Vite Plugin** (`plugins/service-worker-plugin.ts`)
   - Custom plugin that hooks into Vite's build process
   - Automatically collects assets for precaching
   - Injects cache manifest with content hashes
   - Optimizes Service Worker during compilation

2. **Service Worker** (`app/sw.ts`)
   - Implements multiple caching strategies
   - Handles lifecycle events (install, activate, fetch)
   - Manages cache versions and cleanup
   - Supports runtime caching decisions

3. **React Integration**
   - `useServiceWorker` hook for lifecycle management
   - `ServiceWorkerManager` component for UI feedback
   - Automatic registration in `root.tsx`

4. **Utilities** (`app/utils/serviceWorker.ts`)
   - Cache management functions
   - Update detection and handling
   - Status monitoring

## Caching Strategies

### 1. Precache Strategy (Install Phase)

- Static assets cached during Service Worker installation
- Includes JS bundles, CSS, fonts, and critical resources
- Uses content hashes for cache busting
- Defined in `PRECACHE_MANIFEST` (auto-generated by Vite plugin)

### 2. Cache First (Static Assets)

```
Request → Check Cache → Return if found → Fetch & Cache if not
```

- Best for: Images, fonts, static JS/CSS
- Fast response time
- Updates only when cache is missed

### 3. Network First (API Calls)

```
Request → Try Network → Return if success → Fallback to Cache
```

- Best for: API endpoints, dynamic content
- Always tries fresh data first
- Graceful degradation when offline

### 4. Stale While Revalidate (HTML Pages)

```
Request → Return Cache Immediately → Update Cache in Background
```

- Best for: HTML documents, SPA routes
- Instant loading from cache
- Background update for next visit

## Vite Plugin Optimization

The custom Vite plugin (`serviceWorkerPlugin`) provides several build-time optimizations:

### Hooks Used

1. **`configResolved`**
   - Captures final build configuration
   - Determines output directory structure

2. **`generateBundle`**
   - Collects all build artifacts
   - Calculates content hashes for each asset
   - Generates precache manifest automatically
   - Excludes Service Worker itself from precaching

3. **`closeBundle`**
   - Reads Service Worker source template
   - Injects generated precache manifest
   - Writes optimized Service Worker to output
   - Optionally injects registration script into HTML

### Benefits

- ✅ No manual manifest maintenance
- ✅ Automatic cache busting with content hashes
- ✅ Zero configuration for common asset types
- ✅ Build-time validation and error checking
- ✅ Optimized output size

## Installation & Setup

The Service Worker is automatically registered in production builds. No manual setup required.

### Development Mode

Service Worker is **disabled** in development mode to avoid caching issues during hot module replacement.

### Production Mode

Service Worker activates automatically on page load in production builds.

## Usage Examples

### Basic Hook Usage

```tsx
import { useServiceWorker } from "./hooks/useServiceWorker";

function MyComponent() {
  const { status, hasUpdate, skipWaiting } = useServiceWorker();

  return (
    <div>
      {status.isActive ? "🟢 Online" : "⚪ Offline"}

      {hasUpdate && (
        <button onClick={skipWaiting}>
          Update Available - Click to Refresh
        </button>
      )}
    </div>
  );
}
```

### Manual Cache Management

```tsx
import { clearAllCaches, getCacheInfo } from "./utils/serviceWorker";

async function handleClearCache() {
  await clearAllCaches();
  console.log("All caches cleared");
}

async function checkCacheUsage() {
  const info = await getCacheInfo();
  console.log("Cache usage:", info);
}
```

### Programmatic Precaching

```tsx
import { precacheUrls } from "./utils/serviceWorker";

// Precache specific routes
await precacheUrls(["/api/user/profile", "/api/settings"]);
```

## Configuration

### Plugin Options

In `vite.config.ts`:

```typescript
serviceWorkerPlugin({
  swSrc: "./app/sw.ts", // Source file path
  swDest: "./build/client/sw.js", // Output path
  injectRegister: true, // Auto-inject registration
  precachePatterns: [
    // File patterns to precache
    "**/*.{js,css,html,json,png,jpg,jpeg,gif,svg,woff,woff2}",
  ],
});
```

### Service Worker Configuration

Edit `app/sw.ts` to customize:

- Cache names and versions
- URL matching patterns
- Caching strategies per route
- Offline fallback responses

## Lifecycle Events

### Install

- Triggered when new Service Worker is downloaded
- Precaches static assets
- Waits until all resources are cached

### Activate

- Cleans up old cache versions
- Takes control of all pages
- Enables immediate activation with `skipWaiting()`

### Fetch

- Intercepts all network requests
- Applies appropriate caching strategy
- Falls back gracefully on errors

### Update

- Detected when new Service Worker is found
- Enters "waiting" state
- Activates after user refresh or `skipWaiting()`

## Best Practices

### 1. Cache Versioning

Always increment cache names when making breaking changes:

```typescript
const CACHE_NAME = "app-cache-v2"; // Increment version
```

### 2. Selective Caching

Only cache what users need offline:

- Critical JS/CSS bundles
- Essential images and fonts
- Frequently accessed API responses

### 3. Cache Size Management

Monitor and limit cache usage:

```typescript
const info = await getCacheInfo();
if (totalSize > MAX_CACHE_SIZE) {
  await clearOldCaches();
}
```

### 4. Update Notifications

Inform users about available updates:

```tsx
{
  hasUpdate && <UpdateNotification onRefresh={skipWaiting} />;
}
```

### 5. Error Handling

Always handle offline scenarios:

```typescript
try {
  const response = await fetch(request);
  // ...
} catch (error) {
  return offlineFallback;
}
```

## Performance Impact

### Benefits

- ⚡ Faster page loads (cached resources)
- 📱 Offline functionality
- 🔄 Seamless updates
- 💾 Reduced server load

### Considerations

- Initial load includes Service Worker download (~1-5KB)
- Cache storage uses disk space (typically 10-50MB)
- Background updates consume bandwidth
- First visit may be slower (precaching)

### Optimization Tips

1. Keep Service Worker code minimal
2. Use efficient caching strategies per resource type
3. Implement cache expiration for stale data
4. Monitor cache hit rates in analytics

## Troubleshooting

### Service Worker Not Registering

**Check:**

- Running in production mode?
- HTTPS or localhost?
- Browser supports Service Workers?

**Solution:**

```typescript
console.log("SW supported:", "serviceWorker" in navigator);
console.log("Protocol:", window.location.protocol);
```

### Cache Not Updating

**Issue:** Old cached resources persist

**Solution:**

1. Increment cache version in `sw.ts`
2. Clear browser cache manually
3. Use `clearAllCaches()` utility

### Updates Not Detected

**Issue:** New Service Worker doesn't activate

**Solution:**

1. Ensure `skipWaiting()` is called
2. Check browser DevTools for waiting SW
3. Verify `clients.claim()` in activate event

### Precaching Failures

**Issue:** Some assets not cached

**Solution:**

1. Check Vite build output for errors
2. Verify file paths are correct
3. Review `precachePatterns` configuration

## Advanced Features

### Background Sync

Implement background synchronization for offline actions:

```typescript
self.addEventListener("sync", (event) => {
  if (event.tag === "sync-messages") {
    event.waitUntil(syncMessages());
  }
});
```

### Push Notifications

Add push notification support:

```typescript
self.addEventListener("push", (event) => {
  const data = event.data?.json();
  self.registration.showNotification(data.title, {
    body: data.body,
    icon: "/icon.png",
  });
});
```

### Periodic Background Sync

Sync data periodically (Chrome-only):

```typescript
self.addEventListener("periodicsync", (event) => {
  if (event.tag === "update-feed") {
    event.waitUntil(updateFeed());
  }
});
```

## Migration Guide

### From Manual Registration

If you previously had manual Service Worker registration:

**Before:**

```html
<script>
  navigator.serviceWorker.register("/sw.js");
</script>
```

**After:**
Remove manual registration - it's now handled by:

1. Vite plugin auto-injection (in `index.html`)
2. React component registration (in `root.tsx`)

### From Other Plugins

Migrating from `vite-plugin-pwa` or similar:

1. Remove old plugin from `vite.config.ts`
2. Add `serviceWorkerPlugin` configuration
3. Move Service Worker logic to `app/sw.ts`
4. Update import paths in React components
